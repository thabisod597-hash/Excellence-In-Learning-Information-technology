LESSON 1: INTRODUCTION TO DATABASE

1. What is Data?

Data are raw facts that have not yet been processed.

Examples:

a)Thabiso
b)25
c)Male
d)E250
These facts alone do not give much meaning.

2. What is Information?

Information is processed data that has meaning and helps people make decisions.

Example:

a)Name
b)Product
c)Quantity
d)Tom
e)Apples
f)14
This table is information because it tells us something useful.

3. What is a Database?

A database is a collection of related data stored electronically so that it can be accessed, managed, and updated easily.

Examples

a)Student database
b)Hospital database
c)Bank database
d)School database

4. What is a DBMS?

DBMS stands for Database Management System.
It is software used to create, store, update, delete, and manage databases.

Examples of DBMS

a)MySQL
b)Oracle
c)Microsoft SQL Server
d)Microsoft Access

Functions of a DBMS include:

a)Creating databases
b)Storing data
c)Updating data
d)Controlling user access
e)Processing queries

5. Importance of a DBMS

A DBMS:
a)Makes data management easier.
b)Stores data securely.
c)Allows fast searching.
d)Reduces duplicate data.
e)Helps people make better decisions.
f0Allows many users to access data safely.

6. Importance of Good Database Design

A well-designed database:
a)Reduces duplicate data.
b)Prevents errors.
c)Makes data easy to retrieve.
d)Improves system performance.

A poorly designed database:

a)Wastes storage space.
b)Produces incorrect information.
c)Makes maintenance difficult.

7. Historical Development of Databases

Problems with file systems:

a)Duplicate data
b)Difficult to update
c)Slow searching
d)Poor security
e)Databases were introduced to solve these problems.

8. File Terminology

a)Data

Raw facts.

b)Field

A single piece of information.
Example: Name = Tom

c)Record

A group of related fields.
Example: Name: Tom Age: 23 Gender: Male

d)File

A collection of records.

9. Types of Database Systems

a)Single-user Database

Only one person uses it at a time.

b)Multi-user Database

Many users access it simultaneously.

c)Centralized Database

Stored in one location.

d)Distributed Database

Stored in different locations but connected together.

10. Functions of a DBMS

A DBMS performs:

a)Data storage
b)Data security
c)Backup and recovery
d)Multi-user access
e)Data integrity
f)Database communication
g)Data dictionary management

11. Database Models

A database model describes how data is organized.

Main database models:

a)Hierarchical Model
b)Network Model
c)Relational Model
d)Entity-Relationship (ER) Model
e)Object-Oriented Model

12. Hierarchical Database Model

Structure:

Looks like an upside-down tree.
One parent can have many children.
Each child has only one parent.

Advantages

a)Fast
b)Simple
c)Good security

Disadvantages

a)Difficult to modify
b)Complex programming
c)Limited flexibility

13. Network Database Model

One record can have many parents.
More flexible than the hierarchical model.

Advantages

a)Supports many relationships.
b)Better flexibility.

Disadvantages

a)Complex to design.
b)Harder to maintain.

14. Relational Database Model

The most commonly used database model.
Data is stored in tables.

Each table contains:

a)Rows (records)
b)Columns (fields)

Advantages

a)Easy to understand.
b)Easy to manage.
c)Uses SQL.
d)Supports data independence.

Disadvantages

a)Requires more hardware.
b)Poor design can reduce performance.

15. Entity Relationship (ER) Model

Used to design databases before implementation.

Main components:

a)Entity
b)Attribute
c)Relationship

Advantages:

a)Easy to understand.
b)Visual representation.
c)Helps database designers communicate ideas clearly.

16. Object-Oriented Database Model

Stores data as objects.

Advantages:

a)Supports complex data.
b)Reuses code through inheritance.

Disadvantages:

a)Difficult to learn.
b)Requires more computer resources.

Lesson 2

DATA MODELING

1. What is Data Modeling?

Definition: Data modeling is the process of creating a visual representation of data and showing how different pieces of data relate to each other before building a database.

Importance

a)Helps understand business requirements.
b)Reduces errors during database design.
c)Makes the database easier to develop and maintain. d)Improves communication between users and database designers.

2. What is a Data Model?

A data model is a simplified representation of real-world data structures.

It describes:

a)What data will be stored.
b)How data is organized.
c)How different data items are related.

3. Levels (Degrees) of Data Abstraction

A. Conceptual Model

This is the highest level.

Characteristics

a)Gives the overall view of the database.
b)Identifies main entities and relationships.
C)Independent of hardware and software.
d)Represented using an Entity Relationship Diagram (ERD).

Example A school database containing:

a)Student
b)Lecturer
c)Course
d)Department

B. Internal Model

Shows how the database is organized inside the DBMS.

Characteristics

a)Software dependent.
b)Converts the conceptual model into tables.
c)Shows keys and relationships.

C. External Model

Shows what each user sees.

Characteristics

a)Different users see different parts of the database.
b)Improves security.
c)Makes application development easier.

Example

a)Student sees only marks.
b)Lecturer sees marks and student details.
c)Administrator sees everything.

D. Physical Model

Lowest level of abstraction.

Shows:

a)Storage devices
b)Files
c)Indexes
d)Access methods It depends on both hardware and software.

4. Entity Relationship (ER) Model

The ER Model is used to design databases.

It consists of three main components:

a)Entities
b)Attributes
c)Relationships

5. Entity

An entity is a real-world object or thing about which data is stored.

Examples:

a)Student
b)Employee
c)Book
d)Customer
e)Department

In an ER Diagram:

Entity = Rectangle

6. Attributes

Attributes describe an entity.

Example

a)Student
b)StudentID
c)Name
d)Age
e)Gender

Types of Attributes

a)Simple Attribute
Cannot be divided.
Example:
Age
Gender

b)Composite Attribute
Can be divided into smaller parts.
Example:
Address
Street
City
Postal Code

c)Single-Valued Attribute
Has only one value.
Example:
Student Number

d)Multi-Valued Attribute
Has more than one value.
Example:
Phone Numbers
Qualifications

e)Derived Attribute
Calculated from another attribute.
Example:
Age calculated from Date of Birth.

7. Domain

A domain is the set of all possible values an attribute can have.
Example
Gender:
Male
Female
Age:
0–120

8. Primary Key

A primary key uniquely identifies every record in a table.

Characteristics
a)Unique
b)Cannot be NULL
c)One primary key per table
Example
a)Student
b)StudentID Name Age
StudentID is the Primary Key.

9. Relationship

A relationship shows how entities are connected.
Example
a)Student → Enrolls → Course
b)Employee → Works in → Department

10. Connectivity

Connectivity shows how many records are related.
a)One-to-One (1:1)
One record matches one record.
Example: One person → One passport
b)One-to-Many (1:M)
One record relates to many records.
Example: One department → Many employees
c)Many-to-Many (M:N)
Many records relate to many records.
Example: Students register for many courses. Courses have many students.

11. Cardinality

Cardinality specifies the number of entity occurrences participating in a relationship.
Example: One lecturer teaches many students.

12. Relationship Strength

a)Strong Relationship
a)Child depends on parent.
b)Child primary key includes parent's primary key.
b)Weak Relationship
a)Child can exist independently.
b)Child primary key does not include parent's primary key.

13. Participation

a)Mandatory Participation
Entity must participate.
Example: Every employee must belong to a department.
Optional Participation
Entity may participate.
Example: A customer may not place an order.

14. Weak Entity

A weak entity:
a)Cannot exist without another entity.
b)Depends on a parent entity.
c)Uses part of the parent's primary key.
Example: Employee → Dependent
Dependent cannot exist without Employee.

15. Relationship Degree

Shows how many entities participate.
a)Unary Relationship
One entity relates to itself.
Example: Employee supervises Employee.
b)Binary Relationship
Two entities.
Example: Student enrolls in Course.
Ternary Relationship c)Three entities.
Example: Supplier supplies Product to Warehouse.

16. Composite Entity (Bridge Entity)

Used to solve Many-to-Many relationships.
Example
Student ← Enrollment → Course
Enrollment becomes the bridge table.

17. Supertype and Subtype

a)Supertype
General entity.
Example: Employee
b)Subtypes
Specialized entities.
Employee
Lecturer
Administrator
Accountant
The supertype contains common attributes.
Subtypes contain unique attributes.

18. Developing an ER Diagram

Steps
a)Study the business requirements.
b)Identify entities.
c)Identify attributes.
d)Identify primary keys.
e)Identify relationships.
f)Draw the ER Diagram.
Review and improve the design until it is correct.

19. Challenges of Database Design

A good database should balance:
a)Security
b)Performance
c)Data integrity
d)Shared access
e)Fast processing
f)Following design standards

Quick Exam Summary

a)Data Model: Representation of real-world data.
b)Conceptual Model: High-level database view.
c)Internal Model: DBMS view.
d)External Model: User's view.
e)Physical Model: Storage level.
f)Entity: Object stored in the database.
g)Attribute: Property of an entity.
h)Primary Key: Unique identifier.
i)Relationship: Connection between entities.
j)Connectivity: 1:1, 1:M, M:N.
k)Cardinality: Number of related records.
l)Weak Entity: Depends on another entity.
m)Composite Entity: Resolves M:N relationships.
n)Supertype: General entity.
o)Subtype: Specialized version of a supertype.

Lesson 3

Normalization

1. What is Normalization?

Normalization is the process of organizing data in a database to reduce duplication (repeated data) and improve data integrity. It helps create efficient database tables.

Objectives of Normalization

a)Reduce data redundancy (duplicate data)
b)Eliminate data anomalies
c)Improve database consistency
d)Make database maintenance easier

2. Problems Without Normalization

A. Update Anomaly

Occurs when the same information is stored in many places and must be updated multiple times.
Example: If JOB_CLASS = "Manager" is stored in 10 rows, changing it requires updating all 10 rows.
Missing one row causes inconsistency.

B. Insertion Anomaly

Occurs when you cannot add data without adding other unnecessary data.
Example: A new employee cannot be entered unless assigned to a project.

C. Deletion Anomaly

Occurs when deleting one record removes important information.
Example: Deleting an employee may also delete project information.

3. First Normal Form (1NF)

Definition
A table is in 1NF when:
a)Each cell contains only one value.
b)No repeating groups exist.
c)Every row is uniquely identified by a primary key.

Example (Not in 1NF)

StudentID
Name
Subjects
101
John
DB, Java

The Subjects column contains multiple values.
Convert to 1NF
StudentID
Name
Subject
101
John
DB
101
John
Java

1NF Summary

a)Primary key defined
b)No repeating groups
c)Attributes depend on the primary key

4. Functional Dependency

A functional dependency exists when one attribute determines another.
Example
a)StudentID
b)StudentName
c)101
d)John
e)StudentID → StudentName
Meaning: StudentID determines StudentName.

5. Partial Dependency

Occurs when an attribute depends on only part of a composite primary key.
Example
Proj_Num
Emp_Num
Emp_Name
Primary Key = (Proj_Num, Emp_Num)
Emp_Name depends only on Emp_Num, not the whole key.
This is called Partial Dependency.

6. Second Normal Form (2NF)

Definition

A table is in 2NF when:

a)It is already in 1NF.
b)It has no partial dependencies.
Example
Original Table:
Proj_Num
Emp_Num
Emp_Name
Hours
Split into:
PROJECT
Proj_Num
Proj_Name
EMPLOYEE
Emp_Num
Emp_Name
ASSIGN
Proj_Num
Emp_Num
Hours
This removes partial dependencies.

2NF Summary

a)Must be in 1NF
b)No partial dependency
c)Transitive dependency may still exist

7. Transitive Dependency

Occurs when a non-key attribute depends on another non-key attribute.
Example
Emp_Num
Emp_Name
Job_Class
Chg_Hour
Emp_Num → Job_Class
Job_Class → Chg_Hour
Therefore:
Emp_Num → Chg_Hour
This is a transitive dependency.

8. Third Normal Form (3NF)

Definition

A table is in 3NF when:

a)It is already in 2NF.
b)It contains no transitive dependencies.

Conversion to 3NF

Separate JOB information:
EMPLOYEE
Emp_Num
Emp_Name
Job_Class
JOB
Job_Class
Chg_Hour
This removes transitive dependency.

3NF Summary

a)Must be in 2NF
b)No transitive dependency

9. BCNF (Boyce-Codd Normal Form)

BCNF is a stronger version of 3NF.

Rule

Every determinant must be a candidate key.
Simple Exam Definition
BCNF removes special dependency problems that may still exist in 3NF.

10. Fourth Normal Form (4NF)

Rule

A table should not contain multiple independent multivalued attributes.
Example
Student
Hobby
Language
One student may have:
Many hobbies
Many languages
Separate them into:
Student_Hobby
| Student | Hobby |
Student_Language
| Student | Language |
This achieves 4NF.

11. ERD and Normalization

ERD (Entity Relationship Diagram)
Shows:
a)Entities
b)Attributes
c)Relationships

Normalization

Shows:
How attributes should be organized into tables
Both are used together to create a good database design.

Denormalization means intentionally adding some redundancy to improve performance.

Advantages

a)Faster queries
